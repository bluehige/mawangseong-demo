# 01. 코어 시스템 규칙

## 1. 논리 그리드와 렌더링 그리드를 분리한다

Codex는 반드시 **논리 좌표**를 기준으로 맵을 관리해야 한다.

- 논리 좌표: `(gx, gy)`
- 렌더 좌표: `grid_to_world(gx, gy)` 변환 결과

즉, 연결 계산, 이동 가능 여부, 방 배치, 통로 연결, 마스크 계산은 모두 logical grid 기준이다.
쿼터뷰는 **보이는 방식만 다를 뿐**, 연결 계산은 일반 2D 격자와 동일하다.

---

## 2. 마스터 그리드를 고정한다

현재 프로젝트에서 가장 큰 문제 중 하나는 "등급이 오를 때마다 grid 자체를 새로 만들려고 하는 것"이다.
이 방식은 복잡하고 버그가 많다.

따라서 구현은 다음처럼 한다.

- 게임 전체에서 사용할 최대 그리드 크기 `MAX_GRID_W`, `MAX_GRID_H`를 먼저 정한다.
- 예: `20 x 20`
- 처음부터 20x20 배열을 만든다.
- 하지만 처음에는 일부 셀만 `active = true`
- 마왕성 업그레이드 시 더 바깥쪽 셀을 `active = true`로 바꾼다.

플레이어는 "마왕성이 커졌다"고 느끼고,
코드는 기존 좌표를 옮기지 않아도 된다.

---

## 3. 셀의 기본 상태

각 셀은 최소 다음 상태를 가진다.

```json
{
  "gx": 10,
  "gy": 8,
  "active": true,
  "cell_type": "floor",
  "theme": "cave_f",
  "walkable": true,
  "room_id": "room_barracks_01",
  "is_corridor": false,
  "has_socket": false,
  "socket_state": "none",
  "object_id": null,
  "trap_id": null
}
```

### cell_type 종류

- `void` : 아직 마왕성이 확장되지 않은 바깥 영역
- `rock` : 활성 영역 안이지만 건설되지 않은 암반 영역
- `floor` : 바닥 셀
- `wall_anchor` : 벽 또는 외곽 처리용 기준 셀
- `door` : 문이 놓이는 셀
- `socket_marker` : 연결 예정 지점

실제 렌더 시에는 `cell_type`과 주변 상태를 바탕으로 floor / edge / wall / decor를 조합해서 보여준다.

## 4. 레이어 구성

반드시 아래 레이어를 분리한다.

1. `BackgroundVoidLayer`
   - 가장 뒤
   - 거대한 지반/어둠/동굴 벽 배경 이미지
2. `FloorLayer`
   - 실제 바닥 다이아몬드 타일
3. `EdgeSkirtLayer`
   - 바닥이 끝나는 외곽 경사/절단면
4. `BackWallLayer`
   - 뒤쪽 벽, 기둥, 폐쇄된 입구 등
5. `ObjectBackLayer`
   - 큰 오브젝트의 뒷부분
6. `UnitYSortLayer`
   - 몬스터와 적
7. `ObjectFrontLayer`
   - 큰 오브젝트의 앞부분
8. `FrontWallLayer`
   - 화면 아래쪽 전경 벽
9. `FxLayer`
   - 전투 이펙트
10. `UiDebugLayer`
   - 디버그 표시

## 5. 방과 통로는 "셀 집합"이다

방과 통로는 각각 독립 PNG가 아니다.

예:

```json
{
  "blueprint_id": "corridor_straight_3",
  "cells": [[0,0], [1,0], [2,0]],
  "walk_cells": [[0,0], [1,0], [2,0]],
  "socket_entries": {
    "west": [0,0],
    "east": [2,0]
  }
}
```

즉, 통로도 결국 floor cell 여러 개의 집합이다.
연결되기 전과 후는 동일한 PNG를 쓰는 것이 아니라,
소켓 상태와 이웃 상태에 따라 같은 cell도 다른 타일/오브젝트를 쓴다.

---

## 6. 소켓 상태 3단계

소켓은 최소 다음 3상태를 가져야 한다.

- `closed`
  - 아직 연결되지 않았고, 막힌 상태
  - 동굴 벽 / 막힌 문 / 흙벽 / 철문 등으로 보인다
- `open_placeholder`
  - 연결 가능한 위치지만 아직 아무것도 안 붙음
  - 건설 슬롯처럼 보인다
- `connected`
  - 실제 방이나 통로가 붙어 있음
  - 문틀/통로 연결로 보인다

이 3가지를 구분하지 않으면 "연결 전"과 "연결 후" 이미지 변화가 생기지 않는다.

## 7. 배경 이미지는 어떻게 쓰는가

사용자 의견대로 배경 이미지를 미리 만들어두는 방식은 유효하다.
다만 용도는 아래로 제한한다.

### 배경 이미지의 역할

- 전체 동굴 분위기
- 마왕성 밖의 암석/어둠/지반 표현
- 화면에 비어 보이는 부분 채우기
- 카메라 이동 시 타일 바깥 공간의 미관 유지

### 배경 이미지가 하면 안 되는 역할

- walkable 판정
- 방 연결 여부 판정
- 통로가 연결되었는지 여부 판정
- exposed edge 판정

즉, **배경은 뒤에 깔리는 분위기 레이어**이고,
실제 맵의 연결/이동/구조는 반드시 cell 데이터와 타일 규칙으로 처리한다.
