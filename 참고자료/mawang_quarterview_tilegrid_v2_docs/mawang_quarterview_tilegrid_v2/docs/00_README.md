# 마왕님, 마왕성은 누가 지켜요?
# 2D 쿼터뷰 커스텀 맵 생성 규칙 v2

이 문서는 Codex가 현재 제대로 이해하지 못하고 있는 세 가지 핵심 문제를 해결하기 위해 작성되었다.

1. 마왕성 업그레이드에 따라 그리드 칸 수가 늘어나는 규칙
2. 연결 상태에 따라 16개 타일 중 어떤 이미지를 써야 하는지 결정하는 규칙
3. 통로/방이 연결되기 전과 연결된 후의 시각 차이, 그리고 빈 공간을 어떻게 처리할지에 대한 규칙

이 문서의 최종 목표는 다음이다.

- Codex가 **통짜 방 이미지 연결 방식**을 완전히 버리고,
- **가상의 논리 그리드(logical grid)** 위에서 쿼터뷰 타일을 조립하고,
- **바닥/벽/전경/배경을 레이어 분리**해서,
- **마왕성 성장, 방 연결, 미연결 상태, 통로 확장**까지 일관되게 구현하도록 만드는 것.

---

## 가장 중요한 결론

### 결론 1. 그리드는 "실제로 매번 새로 만드는 가변 배열"보다 **고정 최대 그리드 + 등급별 활성/비활성 셀**이 낫다.

플레이어에게는 마왕성이 커지는 것처럼 보이지만, 구현은 훨씬 단순해진다.

### 결론 2. 방 하나는 이미지 1장이 아니다.

방 하나는 아래의 조합이다.

- 여러 개의 logical cell
- 각 cell의 floor tile
- 각 cell의 exposed edge / wall skirt
- 소켓 상태(막힘 / 연결 가능 / 연결됨)
- walkable 여부
- 소품 / 함정 / 문 오브젝트

### 결론 3. 배경 이미지는 "해결책의 일부"다.

사용자 의견처럼 **배경 이미지를 먼저 만들어 두고 그 위에 오브젝트를 얹는 방식**은 맞는 방향이다.
하지만 **배경만으로는 해결되지 않는다.** 이유는 다음과 같다.

- 통로 끝, 막힌 벽, 미연결 입구는 배경만으로 자연스럽게 표현되지 않는다.
- 방이 확장될 때 exposed edge(노출 가장자리)와 filled floor(연결된 바닥)가 달라져야 한다.
- 따라서 배경 이미지는 **void/암석/지반 분위기 레이어**로 사용하고,
  실제 연결/미연결 표현은 **타일 및 오버레이 규칙**으로 처리해야 한다.

---

## Codex가 절대 하면 안 되는 것

1. 구역당 배경 PNG 한 장을 그대로 이어 붙이는 방식
2. 소켓 위치만 보고 직선으로 방을 잇는 방식
3. 통로 연결 전후에 같은 이미지를 재사용하는 방식
4. grid 크기를 늘릴 때 모든 기존 좌표를 다시 재계산하는 방식
5. walkable cell 검증 없이 보이는 이미지 위를 그대로 이동시키는 방식
