# 마왕성 2.0 복구 기획 요청서 — GPT Pro 검토용

> 이 문서는 GPT Pro에게 그대로 전달하는 검토 요청서다. 아래 저장소, 커밋, 실제 공개 빌드와 코드를 먼저 확인한 뒤 한국어로 답변해 달라.

## 0. 이번 요청

현재 마왕성 2.0은 기능 추가를 계속할 단계가 아니다. 기존 1.2보다 전투 판단과 가독성이 퇴행했고, UI 수정과 경로 패치를 반복했지만 체험자가 핵심 플레이를 이해하지 못했다.

저장소와 Git 이력, 실제 전투 코드를 직접 감사한 뒤 다음을 결정할 수 있는 **실행 가능한 복구 마스터 플랜**을 작성해 달라.

- 무엇을 보존하고 무엇을 제거·재구현할 것인가
- 어떤 순서로 고쳐야 같은 퇴행을 반복하지 않는가
- UI, 몬스터 성장, 시설·몬스터 배치, 적 AI와 밸런스를 어떻게 하나의 플레이 루프로 통합할 것인가
- 각 단계가 실제 플레이에서 작동한다고 판정할 객관적 기준은 무엇인가
- 최초 구현 PR 3개는 각각 무엇을 변경하고 무엇을 증명해야 하는가

지금은 코드를 작성해 달라는 요청이 아니다. 먼저 이 저장소에 맞는 수정 순서와 기획을 확정하려는 요청이다.

추상적인 조언은 필요하지 않다. 가능한 모든 판단에 파일, 함수, 커밋, 테스트 또는 실제 동작 근거를 제시하라. 저장소에서 확인할 수 없는 내용은 `가정` 또는 `추가 확인 필요`로 구분하라.

## 2. 2.0을 처음 만든 이유

기존 버전은 기능과 콘텐츠 양에 비해 다음 문제가 있었다.

1. UI에 버튼과 정보가 많지만 신규 사용자는 지금 무엇을 해야 하는지 알기 어려웠다.
2. 앞줄 탱커와 뒷줄 원거리 같은 한 조합으로 대부분의 상황을 해결할 수 있었다.
3. 건물 배치가 침입 과정, 교전 위치, 표적 선택과 결과를 충분히 바꾸지 못했다.
4. 적 종류와 패턴이 달라도 플레이어 대응은 크게 달라지지 않았다.
5. 난이도가 전략적 판단보다 체력과 전투 시간만 늘리는 방향으로 느껴졌다.
6. 의도한 핵심 재미인 `몬스터 성장`, `건물·몬스터 배치`, `다양한 적 패턴 대응` 중 몬스터 성장만 일부 체감됐다.

그래서 2.0은 기존 게임 전체를 바로 교체하는 작업이 아니라, DAY 1~5의 작은 테스트 환경에서 다음 핵심 재미를 먼저 증명하기 위해 시작했다.

- 몬스터 역할과 성장이 전투 방식을 바꾼다.
- 시설과 몬스터 배치가 교전 위치와 전투 결과를 바꾼다.
- 적의 의도를 보고 제한된 전술 명령으로 대응할 수 있다.
- 신규 사용자가 별도 설명 없이 준비에서 전투까지 진행할 수 있다.
- 위 요소가 검증된 이후에만 기존 스토리와 전체 전투 시스템을 이 기준으로 수정한다.

원래 계약상 검증된 1.2 전투 기반은 폐기 대상이 아니라 보존 대상이었다. 여기에는 자동전투, 몬스터 자동 스킬, 적 특수행동, 렌더링, 저장·계측과 함께 1.2 사용자 QA에서 실제 완주로 확인된 경로 정체 복구가 포함돼 있었다.

## 3. 사용자가 정의한 최종 목적

2.0 테스트의 최종 목적은 아래 세 가지다.

### 3.1 UI 최적화 및 단순화

- 신규 사용자가 현재 해야 할 행동을 즉시 이해해야 한다.
- 상시 표시할 정보, 상황별 정보, 선택할 때만 보여 줄 정보를 구분해야 한다.
- 침입 순서 확인, 시설 설치, 몬스터 배치, 방어 시작까지 한 흐름으로 이해돼야 한다.
- 건물마다 설정창이 계속 떠 있거나 의미를 알 수 없는 탭과 버튼이 화면을 차지해서는 안 된다.
- 참고자료의 위치와 정보 계층은 참고할 수 있지만 디자인을 복제해서는 안 된다. 마왕성 배경과 미술 방향에 맞는 완성도 높은 UI여야 한다.

### 3.2 몬스터 성장과 건물 배치 기능 최적화

- 보유 몬스터를 보고 드래그해 유효한 슬롯에 배치할 수 있어야 한다.
- 몬스터 역할, 성장과 배치 위치가 실제 이동, 표적, 생존과 결과를 바꿔야 한다.
- 시설은 단순 수치 보너스가 아니라 교전 위치, 방어 단계, 몬스터 역할 또는 적 대응법을 바꿔야 한다.
- 서로 다른 몬스터 조합과 시설 배치가 실제로 다른 전략과 결과를 만들어야 한다.

### 3.3 적절한 적 AI와 강력함으로 만드는 재미있는 전투 밸런스

- 적은 목적, 역할과 우선순위가 있어야 하며 이유 없이 제자리에서 시간을 보내서는 안 된다.
- 플레이어 몬스터도 역할과 전술 명령을 이해할 수 있는 방식으로 행동해야 한다.
- 적의 강함은 단순 체력 증가나 장기전이 아니라 새로운 판단과 대응을 요구해야 한다.
- 자동전투를 유지하되 집결, 집중 공격, 시설 발동 같은 제한 명령이 필요한 순간과 결과가 명확해야 한다.
- 명령 미사용과 적절한 사용의 전개·피해·결과가 실제로 달라야 한다.

이 세 가지가 검증되기 전에는 기존 스토리와 전체 전투 시스템을 2.0 기준으로 확장하지 않는다.

## 4. 현재 우선하는 게임 구조

다음은 현재 사용자 결정이다. 과거 명세와 충돌한다면 이 요구를 우선해 새 계약으로 정리해야 한다.

1. 침입의 큰 동선과 방 순서는 미리 정해져 있다.
2. 입구에서 마왕성까지 곧바로 도달하는 경로가 있어서는 안 된다.
3. 적은 입구와 복수의 방어 구역을 차례로 돌파해야 한다.
4. 전략 차이는 길을 매번 만드는 것이 아니라, 정해진 구역의 어느 위치에 어떤 시설과 몬스터를 배치하는가에서 발생해야 한다.
5. 준비 화면과 실제 전투는 동일한 공간·방·슬롯·좌표 체계를 사용해야 한다.
6. 자동전투를 유지한다.
7. 개별 유닛을 RTS처럼 직접 이동·공격시키는 방식은 재도입하지 않는다.
8. 소수의 제한 명령은 몬스터 AI보다 명확한 우선권을 가지며 실제 행동과 결과를 바꿔야 한다.
9. 전투 장면은 어두운 분위기를 유지하더라도 유닛, 경로, 공격, 위험 신호와 명령 결과가 명확히 보여야 한다.

과거 명세의 `복수 경로 선택`과 현재의 `고정된 큰 동선`은 충돌한다. 고정된 방 순서 안에서 적 편대, 방 진입 위치, 작전 목표, 교전 우선순위와 시설 대응을 어떻게 다양화할지 새 계약으로 정의해 달라.

## 5. 현재 확인된 실패 증상

- 신규 사용자가 준비 화면에서 첫 행동을 이해하지 못한다.
- 의미를 알기 어려운 탭, 버튼과 상시 설정 패널이 화면을 복잡하게 만든다.
- 건물 설치 과정과 유효 위치가 직관적이지 않다.
- 몬스터 목록, 역할과 드래그 배치가 즉시 이해되지 않는다.
- 침입 경로 표시가 실제 게임 공간보다 별도의 추상 UI처럼 느껴진다.
- 준비 화면에서 정한 전술 공간과 실제 전투 공간이 같은 맵인지 확신하기 어렵다.
- 전투 중 적과 몬스터의 이동·공격·대기 이유가 잘 보이지 않는다.
- 적이 움직이지 않고 방에서 오래 정지하는 것처럼 보인다.
- 공병, 도둑, 돌파, 전열 보호와 후열 같은 적 역할 차이가 행동으로 충분히 드러나지 않는다.
- 플레이어 몬스터가 왜 이동하거나 배치 방으로 돌아가는지 이해하기 어렵다.
- 집결·후퇴 명령이 UI에서는 발동해도 실제 이동에 반영되지 않을 수 있다.
- 명령 없이도 전투가 자동으로 해결돼 전략 개입의 필요성이 약하다.
- 체험자 피드백은 핵심 전투와 UX의 `NO-GO` 신호였지만, UI와 경로 패치를 추가하는 방향으로 계속 진행됐다.

## 6. 현재 원인 감사 결과

아래는 저장소와 Git 이력을 비교한 진단이다. 그대로 받아들이지 말고 코드를 직접 검토해 `동의 / 부분 동의 / 반대 / 추가 확인 필요`로 판정해 달라.

### 6.1 하나의 전투가 아니라 두 모델이 생겼다

v20 준비 섹션은 별도 ID를 유지한 채 기존 runtime room으로 변환된다. 현재 네 배치 섹션은 서로 다른 runtime room에 대응하지만, 전투 route node 일부는 alias 변환 과정에서 같은 runtime room으로 축약된다. 과거 구현에서는 준비에서 계산한 경로 전체가 아니라 마지막 목적 방을 기존 경로 탐색에 넘기기도 했다.

따라서 준비 보드, route node와 runtime room이라는 복수 표현 계층이 존재하고 그 연결이 하나의 지도 계약으로 보장되지 않는다.

### 6.2 새 v20 로직이 기존 AI 앞에서 조기 반환한다

기존 AI 전체가 사라진 것은 아니다. 몬스터 측 v20 역할 이동은 생존 후퇴·왕좌 긴급 방어·일부 고유 행동·자동 스킬 뒤에 있지만, 도둑 추격·아군 호위·금고 수호·입구 봉쇄·함정 유도·총공격·일반 방어보다 먼저 반환할 수 있다. 적 측 checkpoint도 일부 특수행동과 공병 처리 뒤, 도둑 전용 행동과 일반 교전·목표 이동 앞에서 반환한다.

따라서 기존 AI가 삭제된 것이 아니라 삽입 지점 아래의 일부 기존 분기까지 실전에서 도달하지 못하거나 우선순위가 약화돼 더 단순해진 것으로 판단된다.

### 6.3 Phase 11S가 역할·경로 정책을 사문화했다

`afa2cc5`에서 fixed route가 성공하면 역할별 route policy 계산 전에 반환한다.

실전 몬스터 문맥에는 배치 방이 `manual_anchor_node`로 항상 들어가며, 역할 서비스는 이 값이 있으면 부상 아군 보호, 표적 추격, 위험 대응과 후방 위치 판단보다 먼저 반환한다. 배치가 전술 기준점이 아니라 하드 리시처럼 작동한다.

### 6.4 Phase 11T가 눈에 보이는 정지를 직접 추가했다

`d1829eb`은 route node가 부여된 비시설 목표 적이 수비대가 없는 비최종 checkpoint에 도달하면 navigation을 중단하고 1.25초 기다리게 한다. 이 분기는 기존 도둑·일반 목표 이동보다 먼저 실행된다.

별도 돌파 애니메이션이나 공간 상호작용 없이 navigation 정지와 `관문 돌파 n.n초` 상태 문자열만 표시되므로, 전장에서는 아무것도 하지 않는 것처럼 보일 수 있다. 공통 checkpoint 상태기는 그 뒤에 있는 도둑·일반 목표 분기를 약화한다.

### 6.5 전술 명령과 기존 공격 루프가 충돌한다

집결·후퇴가 경로를 만들어도 같은 프레임의 자동 공격이 `stop_navigation()`을 호출해 이동을 취소할 수 있다. 이동 명령 함수는 현재 유닛 인자를 사용하지 않아 전군에 같은 방식으로 적용된다.

명령 테스트는 상태, 자원과 신호를 확인하지만 실제 유닛이 명령 목적지까지 이동하는지는 검증하지 않는다.

### 6.6 테스트가 실제 전투를 검증하지 않았다

여러 테스트가 다음 방식에 의존한다.

- 경로·대응 태그 문자열 생성 확인
- 서비스 점수와 예상 결과 계산
- 테스트 코드가 유닛 좌표와 방 상태를 직접 변경
- 전투 시작 직후 결과 함수를 직접 호출
- 버튼 신호, 드래그와 콘솔 오류 유무 확인

이 방식으로는 실제 물리 프레임에서 유닛이 입구부터 왕좌까지 이동하는지, 공격하는지, 정체를 복구하는지, 역할별로 다르게 판단하는지, 명령이 결과를 바꾸는지 확인할 수 없다.

1.2 사용자 QA는 0.75초간 진행이 없으면 오래된 경로점을 버리고 재탐색하는 정체 복구와 실제 DAY 1 완주를 확인했다. 현재 checkpoint는 의도적으로 이동을 중단하므로 기존 복구가 해결할 수 없다.

### 6.7 페이즈 완료 판정이 제품 검증보다 빨랐다

Git 기록상 Phase 0 명세부터 Phase 10 기능 통합까지 약 2시간 11분, Phase 1부터 Phase 10까지 약 1시간 46분이었다. 속도 자체가 버그의 증명은 아니지만, 서비스·UI 계약 확인을 실제 플레이 통합 PASS로 오인하기 쉬운 구조와 결합됐다.

원래 6~10명 블라인드 테스트 게이트가 완료되지 않았는데도 Phase 11R·11S·11T 패치를 계속 쌓았다. `TARGETED_PASS`를 재미, 직관성, 실전 AI 검증처럼 취급한 것이 핵심 프로세스 실패다.

- 명령 미사용과 적절한 사용이 실제로 다른가
- 모든 적 역할이 의도한 이동과 특수행동을 수행하는가
- 왕좌 목표 적이 입구와 지정된 모든 방어 구역을 거쳐 왕좌까지 스스로 이동하는가
- 보물·시설 목표 적은 예고된 역할별 목표 경로를 따라 이동하고 임무를 수행하는가
- x1과 x3에서 정체와 상태 순서가 깨지지 않는가
- 일정 시간 진행하지 않을 때 자동 복구되는가
- 준비 배치와 전투 위치가 1:1로 일치하는가
- 1280×720에서 UI가 겹치지 않고 조작 가능한가
- 패배·재도전·저장·불러오기 후 배치와 성장이 일관적인가
- 6~10명 초보 블라인드 테스트에서 외부 도움 없이 DAY 1을 완료하는가

다음만으로는 통과시키지 마라.

- 테스트가 좌표나 방 상태를 직접 순간이동
- 전투 결과 함수를 직접 호출
- 메타데이터나 문자열 태그 생성만 확인
- 버튼 신호만으로 명령 성공 판정
- 콘솔 오류가 없다는 이유만으로 플레이 성공 판정

테스트 표 열:

`기능 | 자동 단위 테스트 | 실제 전투 통합 테스트 | 수동 플레이 | 관측 지표 | 합격 기준 | 실패 시 차단할 다음 단계`

### 9.11 최종 수용 기준

2.0을 기존 스토리와 전체 전투 시스템 개편의 기준으로 쓸 수 있는지 판정할 체크리스트를 작성하라. 최소한 다음을 포함하라.

- 신규 사용자가 외부 설명 없이 첫 준비와 전투 시작
- 준비와 전투의 방·슬롯·배치가 완전히 일치
- 모든 적과 몬스터가 실제 좌표에서 목적 있는 행동 수행
- 장시간 무행동이 없거나 자동 복구
- 최소 3개 배치가 다른 전개 또는 결과 생성
- 최소 2개 성장 빌드가 다른 유효 전략 생성
- DAY 2~5의 각 주요 특수 패턴에 최소 2개 대응 존재
- 제한 명령의 사용 여부·시점이 전개 또는 결과 변경
- 한 고정 조합으로 DAY 1~5 무변경 완주 불가
- 난이도 증가가 단순 전투 시간 증가에 머물지 않음
- 실제 블라인드 테스트 기준 통과
- 하나라도 실패하면 기존 스토리 확장 시작 금지

## 10. 답변 형식

아래 순서를 유지하라.

### 1. 집행 요약

- 현재 상태 한 문장 판정
- 가장 중요한 원인 5개
- `현재 2.0 수리`, `1.2 기준 재통합`, `혼합 복구` 중 한 가지 권고와 이유
- 가장 큰 위험

### 2. 감사 결과 검증표

`현재 진단 | 판정 | 코드·커밋 근거 | 실제 영향 | 정정 또는 추가 설명`

### 3. 요구 충돌과 결정

특히 `과거 복수 경로`와 `현재 고정된 큰 동선`을 어떻게 통합할지 명시하라.

### 4. 목표 아키텍처

- Mermaid 구조도
- 데이터·상태 소유권 표
- 준비와 전투의 단일 지도
- 몬스터 AI와 적 AI 우선순위

### 5. 보존·재작업·삭제 결정표

지정한 대상과 열을 빠짐없이 포함하라.

### 6. 페이즈별 복구 계획

`순서 | 목적 | 사용자 변화 | 코드 범위 | 선행조건 | 제거 | 구현 | 검증 | 완료 게이트 | 롤백 기준`

각 페이즈는 다음 단계와 독립적으로 완료 판정할 수 있을 만큼 작게 나눠라.

### 7. 제품 명세

1. UI/UX
2. 몬스터 성장
3. 시설·몬스터 배치
4. 몬스터 AI
5. 적 AI
6. 전투 밸런스

각 항목에 사용자 행동, 시스템 규칙, 피드백, 실패 상태와 합격 조건을 포함하라.

### 8. 테스트 매트릭스

실제 물리 프레임 검증과 블라인드 테스트를 포함하라.

### 9. 최종 수용 체크리스트

각 항목을 자동·수동·사용자 테스트 중 무엇으로 판정할지도 적어라.

### 10. 최초 구현 PR 3개

각 PR에 다음을 포함하라.

- 정확한 목적
- 수정 파일 또는 모듈
- 제거할 충돌 로직
- 추가할 테스트
- 눈으로 확인할 결과
- 병합 조건
- 다음 PR이 의존하는 산출물

### 11. 남은 의사결정

저장소만으로 결정할 수 없는 질문만 최대 5개 제시하라. 가능한 부분은 권고 기본값을 먼저 제시하고 질문으로 미루지 마라.

## 11. 답변 시 금지사항

- `UX를 개선한다`, `AI를 고도화한다`, `밸런스를 조절한다` 같은 실행 기준 없는 표현
- UI 색상과 카드 배치만 바꾸는 처방
- 기존 대형 컨트롤러에 조건문과 조기 반환을 계속 추가하는 방식
- 준비 지도와 전투 지도를 별도로 유지하면서 동기화만 강화하는 처방
- 메타데이터와 예상 점수 계산을 실제 전투 검증으로 간주
- 테스트에서 유닛을 순간이동시키고 경로가 검증됐다고 판정
- 자동 테스트 통과를 재미 검증으로 간주
- 플레이어 피드백 게이트가 실패했는데 다음 페이즈 진행
- 근거 없이 전체 재작성 또는 전체 현행 유지 권고
- 1.2의 검증된 행동을 동등한 실전 테스트 없이 제거

최종 답변은 개발자가 그대로 작업 순서와 PR 단위로 사용할 수 있을 정도로 구체적이어야 한다.
