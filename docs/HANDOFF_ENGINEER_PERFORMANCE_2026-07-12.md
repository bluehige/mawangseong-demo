# 공병 등장·시설 무력화 성능 수정 핸드오프

작성일: 2026-07-12

## 결론

공병 등장 때 보이던 끊김은 검수 도구만의 현상이 아니라 실제 플레이 코드의 성능 문제였다. 공병 한 명이 나타날 때 전투 로그가 연속으로 생기면서 HUD 전체를 두 번 다시 만들고, 공병 애니메이션 16장을 그 순간 읽어 묶었으며, 시설 무력화가 시작된 뒤에는 정적인 마왕성 지도 전체를 매 프레임 다시 그렸다.

세 병목을 모두 수정했고, DAY 20 공병 등장과 10초 시설 무력화·자동 복구를 포함한 전체 검수를 완료했다.

## 수정 전후 계측

| 항목 | 수정 전 | 수정 후 | 결과 |
|---|---:|---:|---:|
| 첫 공병 소환 | 평균 63.48ms | 0.893ms | 약 98.6% 감소 |
| 두 번째 공병 소환 | 평균 46.82ms | 1.202ms | 약 97.4% 감소 |
| 시설 무력화 중 0.5초 맵 다시 그리기 | 30회 | 4회 | 약 86.7% 감소 |
| 전투 로그 1줄 추가 | 화면 전환 신호 재발행·HUD 전체 교체 | 기존 HUD 노드 유지·로그 글자만 갱신 | 전체 재생성 제거 |

60FPS 게임은 한 프레임을 약 16.67ms 안에 처리해야 한다. 수정 전 첫 공병 소환은 이 예산의 약 3.8배였지만, 수정 후에는 1ms 안팎이다.

## 구현 내용

- `Unit.gd`
  - 같은 캐릭터의 `SpriteFrames` 애니메이션 묶음을 한 번만 만들고 모든 개체가 공유하도록 캐시했다.
- `CombatSceneController.gd`
  - 해당 전투에 등장할 적 애니메이션을 전투 화면 전환 전에 미리 준비한다.
  - HUD의 숫자·상태 표시는 초당 10회 제자리 갱신한다.
  - 일시정지 상태에서도 선택 유닛 버튼과 상태 표시는 갱신한다.
- `HUDController.gd`
  - 전투 로그, 자원, 마왕성 체력, 전장 상태, 선택 유닛 상태를 기존 UI 노드를 없애지 않고 글자와 막대만 바꾸도록 수정했다.
  - 선택 몬스터가 쓰러졌을 때 선택 패널과 하단 `직접 조종` 버튼이 함께 비활성화된다.
- `GameRoot.gd`
  - 전투 로그마다 전투 화면 전체를 다시 만드는 호출을 제거했다.
  - 시설 무력화 강조 표시는 실제 시간 기준 초당 최대 10회만 전체 맵 다시 그리기를 요청한다.
- `QuarterDungeonRenderer.gd`
  - 변하지 않는 타일 격자 계산 결과를 재사용하고, 구역 배치나 마왕성 단계가 바뀔 때만 캐시를 비운다.
- `EngineerPerformanceSmokeTest.gd`
  - HUD 노드 유지, 로그 즉시 표시, 공병 애니메이션 공유, 공병 소환 16ms 이내, 무력화 맵 갱신 1~7회를 자동 검사한다.
  - Quick·Full 핵심 통합 검증에 정식 등록했다.

## 검수 결과

- 전용 공병 성능 검사: 9/9 PASS
- 핵심 통합 검증 Quick: 6/6 PASS
- 핵심 통합 검증 Full: 13/13 PASS
- `DemoSmokeTest`: DAY 20 공병 목표 선택, 위협 표시, 시설 무력화, 10초 자동 복구 PASS
- `QuarterModuleSmokeTest`: Stage 01~04 구역·건물·렌더링 PASS
- DAY 20 무력화 화면과 1366 해상도 전투 화면을 직접 확인했으며, HUD 잘림·겹침·누락 없음

첫 Full 실행에서는 DAY 3 자동 전투 시간이 한 번 60.2초로 튀어 12/13이었으나 같은 시나리오를 세 번 다시 실행했을 때 모두 34.3초, DAY 1~3 묶음에서는 36.2초였다. 시스템 오류가 아닌 장시간 검수의 단발 편차로 판단했고, 두 번째 Full 실행에서는 DAY 3이 35.2초로 정상 범위에 들어와 13/13 통과했다.

최신 전체 보고서: `tmp/core_verification/latest.md`

## 남은 비차단 관찰

DAY 20 전투 시작 전환은 차가운 첫 실행 기준 약 181~203ms가 걸린다. 이는 예정된 여러 적의 애니메이션을 전투 중간이 아니라 시작 전환에서 미리 준비하는 비용이다. 실제 전투 중 공병 등장 끊김은 2ms 미만으로 제거됐으므로 이번 결함은 해결됐지만, 추후 전투 시작 로딩 화면이나 백그라운드 준비 작업을 도입할 때 이 전환 시간도 더 줄일 수 있다.
