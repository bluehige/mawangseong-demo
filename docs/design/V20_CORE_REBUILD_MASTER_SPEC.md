## 1. 이 문서가 고정하는 결정

마왕성 2.0은 1.2.1에 기능을 더 얹는 패치가 아니다. 1.2.1 공개판과 불변 태그를 보존하고, 최신 `main`에서 DAY 1~5 PC 버티컬 슬라이스를 다시 만드는 제품 재설계다.

플레이어가 반복해서 판단할 핵심은 다음 세 가지다.

1. 몬스터의 역할 성장을 선택한다.
2. 시설과 몬스터를 배치해 적의 경로와 교전 위치를 바꾼다.
3. 자동 전투 중 제한된 전술 명령으로 예고된 적 패턴에 개입한다.

2.0의 하루 루프는 아래 다섯 단계로 고정한다.

> 정찰 → 건설 → 몬스터 배치 → 자동 전투와 제한 명령 → 결산·성장

`던전키퍼`와 `둥지 짓는 드래곤`에서 가져올 것은 외형이 아니라 던전 자체가 이동, 목표, 교전 위치를 바꾸는 게임 규칙이라는 구조다.

## 2. 기준 저장소에서 직접 확인한 사실

공유 대화의 결론은 방향으로 사용하되, 아래 내용은 2026-07-21에 기준 SHA를 다시 확인한 결과다.

### 2.1 UI 정보가 행동 우선순위보다 기능 목록을 따른다

- `scripts/game/CombatSceneController.gd:247`의 PC 전투 UI는 방 목록, 유닛 현황, 전투 로그, 선택 유닛, 명령, 속도 패널을 모두 동시에 만든다.
- `scripts/ui/HUDController.gd:463`의 선택 유닛 패널은 `370×756px`이며, 자동 전투에서 직접 조작할 수 없는 유닛의 공격력, 방어력, 공격 간격, 방, 상태와 스킬을 항상 표시한다.
- `scripts/ui/HUDController.gd:510`은 전역 지침 3개와 방 지침 3개를 같은 전투 명령 패널에 둔다.
- `scripts/ui/HUDController.gd:530`은 x1, x1.5, x2, x3, 일시정지를 폭 74px 세로 패널에 둔다.
- 관리 UI도 `scripts/game/ManagementSceneController.gd:14`에서 방 목록, 레이아웃 편집, 캠페인 안내, 370px 선택 방 패널, 건설, 몬스터, 전투 시작, 원정·전초기지·상층·연대기·합동기 진입을 한 화면에 조건부로 누적한다.

### 2.2 적 패턴과 현재 입력 계약이 충돌한다

- 1.2는 개별 이동, 직접 공격, 수동 스킬을 제거하고 지침 기반 자동 AI로 전환했다. 근거는 `docs/design/DIRECTIVE_COMBAT_INPUT_SIMPLIFICATION_PLAN_2026-07-16.md`다.
- 그러나 `scripts/game/CombatSceneController.gd:562`는 셀렌의 빛나는 바닥 밖으로 몬스터를 이동시키라고 안내한다.
- `data/regular_version/update3/enemies.json`에는 산성 원 밖으로 직접 이동, 액티브 스킬 사용 조절, 시전 중단 같은 대응 문구가 남아 있다.
- 보호막 시간 제한, 특정 적 집중, 정화·중단 같은 실제 패턴은 존재하지만 플레이어가 그 순간 의도적으로 실행할 공통 입력 계약은 없다.

### 2.3 던전 경로가 전략 보드 계약을 충족하지 않는다

- `data/rooms.json`은 입구가 가시 복도로 들어가고, 가시 복도가 병영·회복·보물·왕좌·건설 슬롯을 연결하는 허브 구조다.
- 시작 상태의 변경 가능한 건설 슬롯은 `slot_01` 하나다.
- `scripts/map/RoomGraph.gd:25`의 `path_between()`은 간선 수만 보는 BFS다. 경로 비용, 문 상태, 시설, 위험도, 적 역할과 목표별 선호가 없다.
- 시설은 실효성 계측이 존재하지만 시작 구조에서는 같은 허브에 수렴하므로, 배치가 적의 길과 교전 위치를 충분히 바꾸지 못한다.

### 2.4 초반 난이도가 수치와 대기로 먼저 증가한다

- `scripts/combat/WaveManager.gd`는 수, 생성 지연, 생성 간격, HP·공격·방어 배율을 중심으로 일정을 만든다.
- `data/waves.json`의 DAY 1 탐험가 HP 배율은 2.1, DAY 2 탐험가 HP 배율은 4.6이다.
- DAY 2는 도둑을 18초 뒤에, DAY 3은 수련생 용사를 18초 뒤에 추가한다.
- 2.0에서는 같은 해결법을 더 오래 반복시키는 대신 목표 분리, 경로 분리, 카운터 적, 예산 제한, 짧아지는 예고 시간으로 판단 수를 늘린다.

### 2.5 이미 존재하는 기반은 버리지 않는다

공유 답변과 달리 현재 저장소에는 선택 차이 계측이 이미 있다.

- `tools/BalanceSimulation.gd`는 시설 선택 A/B와 역할별 기여를 검사한다.
- `tools/DayOneToThreeProxySummary.gd`와 `tools/DayThreeSeedTrial.gd`는 DAY 1~3 자동 대리와 다중 시작값 비교를 제공한다.
- `battle_contribution_stats`는 피해, 흡수, 마무리, 시설 기여를 결산 원본으로 기록한다.
- `data/specializations.json`에는 슬라임, 고블린, 임프 각각 두 개의 조기 특화가 이미 있다.
- UI와 전투 코드는 `ManagementSceneController.gd`, `CombatSceneController.gd`로 일부 분리돼 있다. 다만 `GameRoot.gd`는 여전히 12,008줄이며 상태와 연결 책임이 크다.

따라서 2.0은 이 기반을 삭제하고 처음부터 만드는 작업이 아니다. 검증된 저장, 렌더, 전투, 자산과 계측을 보존하고, 플레이어 결정이 모이는 경로·시설·명령·Encounter 계약을 새 서비스와 데이터 경계로 교체한다.

## 3. 제품 성공 조건

### 3.1 플레이어 경험

- 새 게임 후 90초 안에 첫 의미 있는 배치 또는 시설 선택을 한다.
- DAY 1은 `앞 슬라임 + 뒤 원거리`가 올바른 기본 학습법이다.
- 같은 배치를 수정하지 않고 DAY 5까지 완주할 수 없다.
- DAY 2~5의 각 특수 패턴에는 이해 가능한 대응이 최소 두 개 있다.
- 패배 결산만 보고도 무너진 목표, 경로, 시설과 대응 실패를 한 문장으로 설명할 수 있다.
- 재도전은 이전 배치를 보존하고 한두 개의 선택을 즉시 수정할 수 있다.

### 3.2 전략 차이

- 최소 세 개의 유효 배치가 서로 다른 적 경로 또는 첫 주요 교전 위치를 만든다.
- 시설마다 강점, 명시적 카운터, 몬스터 역할 시너지, 결산 지표가 있다.
- 슬라임·고블린·임프의 각 두 특화가 표기만이 아니라 이동, 표적, 시설 시너지를 바꾼다.
- 제한 전술 명령을 사용한 결과와 사용하지 않은 결과가 대응 성공, 목표 피해 또는 교전 위치에서 달라진다.
- 어려워질수록 판단과 우선순위가 늘고, 평균 전투 시간이 숫자 배율 때문에 일방적으로 늘지 않는다.

### 3.3 UI 계약

- PC 관리 화면의 상시 주 행동은 최대 5개다.
- PC 전투 화면의 전술 명령은 최대 4개다.
- 1280×720 이상에서 상시 HUD와 컨텍스트 드로어가 전장을 가리지 않는다.
- 보이지 않는 기능을 툴팁으로 설명해 해결하지 않는다. 현재 할 수 있는 행동과 결과를 화면 구조가 먼저 드러낸다.

### 3.4 판매 가능성 게이트

DAY 1~5 폴리시 뒤 6~10명의 첫 플레이 관찰을 진행한다. 아래를 모두 만족하기 전에는 DAY 6~30 이식을 시작하지 않는다.

- 참가자 80% 이상이 외부 설명 없이 DAY 1을 완료한다.
- 참가자 70% 이상이 DAY 3까지 시설 또는 배치가 결과를 바꿨다고 자기 말로 설명한다.
- 참가자 70% 이상이 패배 뒤 무엇을 바꿀지 한 가지 이상 말하고 재도전한다.
- 재미, 이해도, 피로도는 자동 대리 결과로 PASS 처리하지 않는다.

## 4. 2.0 DAY 1~5 Encounter 계약

| DAY | 플레이어가 푸는 문제 | 적 구성과 행동 | 배워야 할 대응 | 고정 배치 방지 장치 |
|---:|---|---|---|---|
| 1 | 한 길의 정면 침입 | 탐험가가 왕좌를 향해 한 경로로 진입 | 슬라임 길목 방어, 임프 후방 화력, 전투 읽기 | 없음. 기본 조합이 성공해야 함 |

| 4 | 보호받는 후열 | 방패병이 전열을 만들고 대마법 궁수가 후방에서 압박 | 감시 초소, 임프 포병, 집중 명령, 우회 교전 | 입구 몸빵만으로는 후열을 끊지 못함 |
| 5 | 전열 돌파와 2단계 변화 | 수련생 용사가 첫 방어선을 대시로 넘고 증원 또는 목표 경로를 변경 | 후퇴선, 회복, 집결, 예비 시설 | 한 줄 전선만 준비하면 후방이 붕괴 |

각 Encounter는 다음을 데이터로 선언해야 한다.

- 적의 목표와 예상 경로
- 특수 행동의 예고, 해소 조건, 실패 결과
- 최소 두 개의 플레이어 대응 태그
- 결산에 남길 성공·실패 지표
- 전투 시간 목표 범위와 숫자 배율 상한

## 5. 관리 화면 정보 구조

### 5.1 항상 보이는 항목

1. 오늘의 침입 정보
2. 현재 건설 자원과 명령력 준비치
3. 중앙 던전 전략 보드
4. `방어 시작` 주 버튼

설정과 뒤로가기는 공통 셸에 남길 수 있으나 하루 준비의 주 행동 수에는 포함하지 않는다.

### 5.2 컨텍스트에서만 보이는 항목

- 방 선택: 시설 역할, 연결 경로, 배치 정원, 설치·교체
- 몬스터 선택: 역할, 특화, 현재 배치, 드래그 핸들
- 경로 선택: 예상 침입자와 비용 변화
- 고급 정보: 세부 능력치와 과거 로그

연대기, 원정, 의회, 전초기지, 상층, 합동기, 자유 맵 편집은 삭제하지 않는다. DAY 1~5 핵심 준비 화면에서는 숨기고 2.0 이식 단계 전까지 진입 경로를 만들지 않는다.

### 5.3 건설과 배치 입력

신규 시설 설치는 아래 두 동작으로 끝낸다.

> 빈 슬롯 클릭 → 유효 시설 클릭 → 즉시 설치

- 신규 설치에는 별도 확정을 요구하지 않는다.
- 설치 직후 한 단계 Undo를 제공한다.
- 기존 시설의 파괴적 교체만 `시설 선택 → 교체 결과 확인 → 확정`의 세 동작을 사용한다.
- 유일 시설 이동, 자원 손실, 배치 몬스터 강제 이동은 확인 화면에서 명시한다.
- 몬스터는 초상을 방으로 드래그한다.
- 접근성 대체 입력은 `몬스터 클릭 → 방 클릭`이다.

## 6. 전투 화면과 제한 전술 명령

### 6.1 항상 보이는 항목

- 왕좌 또는 현재 핵심 목표 HP
- Encounter 단계와 남은 목표
- 다음 특수 패턴과 예고 시간
- 최대 3개의 기본 전술 명령과 최대 1개의 Encounter 비상 명령
- 속도와 일시정지

방 목록, 전체 로그, 대형 유닛 상세, 전체 지침 3개, 방 지침 3개는 기본 HUD에서 제거한다. 유닛·방 클릭 또는 일시정지 시 오른쪽 컨텍스트 드로어에서 확인한다.

### 6.2 기존 지침의 새 위치

| 기존 입력 | 2.0 위치 |
|---|---|
| 사수 / 총공격 / 생존 우선 | 전투 전 AI 교리 |
| 입구 봉쇄 / 함정 유도 / 후퇴선 유지 | 방 또는 몬스터 역할 설정 |
| 직접 이동 / 직접 공격 / 수동 스킬 | 다시 도입하지 않음 |

### 6.3 전투 중 공통 명령

| 명령 | 대상 | 의도 | 제한 |
|---|---|---|---|
| 집결 | 방 또는 집결점 | 자동 AI의 현재 우선순위를 잠시 덮고 전선을 재형성 | 명령력과 재사용 대기시간 |
| 집중 | 예고 중인 적 또는 적 집단 | 지정 시간 동안 표적 우선순위를 올려 시전·보호막·공병을 끊음 | 단일 활성 표적, 명령력 |
| 시설 발동 | 활성 시설 | 시설의 수동 결정 순간을 만들고 경로·차단·회복 효과를 강화 | 시설별 충전 또는 1회 사용 |
| 비상 명령 | Encounter가 노출한 상황 | 후퇴, 봉쇄 해제, 정화 등 해당 패턴 전용 대응 | 항상 보이지 않으며 최대 1개 |

명령 세부 수치는 Phase 7에서 정한다. Phase 0에서는 명령이 결과를 바꿀 수 있고 스팸할 수 없어야 한다는 계약만 고정한다.

## 7. 전략 경로 계약

DAY 1~5 던전은 최소 두 개의 침입 가능 경로와 두 개 이상의 방어선 깊이를 가진다.

경로 선택은 단순 BFS가 아니라 다음 값을 포함한 가중 비용으로 계산한다.

```text
total_cost = base_edge_cost
           + door_state_cost
           + facility_route_cost
           + temporary_hazard_cost
           + enemy_role_preference
           + goal_preference
```

- 문과 바리케이드는 통과 가능 여부 또는 큰 비용을 만든다.
- 보물·미끼는 일부 적에게 목표 선호를 만든다.
- 시설 발동은 짧은 시간 경로 비용을 바꿀 수 있다.
- 적 역할마다 `왕좌`, `보물`, `시설`, `가장 약한 목표` 중 목표 계약을 가진다.
- 같은 조건과 seed는 같은 경로와 결과를 재현해야 한다.
- 경로 변경은 전장에 미리 보이며, 숨겨진 랜덤 전환으로 플레이어를 벌하지 않는다.

## 8. 시설과 몬스터 역할 계약

### 8.1 DAY 1~5 시설 역할

| 시설 역할 | 이동·교전 영향 | 강점 | 카운터 | 대표 시너지 | 필수 결산 지표 |
|---|---|---|---|---|---|
| 바리케이드/문 | 간선 차단 또는 비용 증가 | 진입 지연, 전선 형성 | 공병, 돌파 대시 | 성문 파수 슬라임 | 지연 시간, 우회 수, 흡수 피해 |
| 병영 | 지정 방을 주 교전 거점으로 강화 | 전열 화력·생존 | 후열 사격, 우회 | 성문 파수·마무리 칼날 | 강화 피해, 감소 피해, 교전 시간 |
| 미끼 보물실 | 도둑 목표와 경로를 변경 | 분리 유도 | 다중 도둑, 무방비 약탈 | 도둑 사냥꾼 고블린 | 유인 수, 도난, 추격 거리 |
| 감시 초소 | 경로 예고와 영향 범위의 적 노출·둔화 | 후열·공병 대응 | 무력화, 사각 경로 | 장거리 화염술 임프 | 노출 시간, 둔화, 추가 피해 |
| 회복 둥지 | 후퇴선과 재집결 위치를 형성 | 장기전, 2선 복귀 | 돌파, 회복 억제 | 구조 점액 슬라임 | 회복량, 복귀 수, 생존 시간 |

기존 그래픽과 시설 구현을 우선 재사용한다. 바리케이드처럼 새 논리가 필요한 항목도 Phase 4~5의 플레이 검증 전에는 신규 최종 그래픽을 만들지 않는다.

### 8.2 기존 조기 특화 재사용

| 몬스터 | 빌드 A | 빌드 B | 2.0 연결 기준 |
|---|---|---|---|
| 슬라임 | 성문 파수 | 구조 점액 | 차단 전선 고정 vs 부상 아군·후퇴선 호위 |
| 고블린 | 도둑 사냥꾼 | 마무리 칼날 | 목표 추격 vs 부상 표적 제거 |
| 임프 | 장거리 화염술 | 함정 화염술 | 후열 집중 vs 경로·함정 제어 |

특화 이름과 수치가 이미 있으므로 Phase 6은 새 선택지를 늘리는 작업이 아니다. 각 특화가 weighted path, 시설, 집중 명령과 실제로 다른 행동을 만드는지 연결하고 검증한다.

## 10. 저장과 호환성

- `v1.2.0`과 `v1.2.1` 태그, Release, 저장 계약은 변경하지 않는다.
- 2.0 개발 빌드는 기존 `campaign_save_v1.json`을 덮어쓰지 않는다.
- Phase 10 전까지 2.0은 별도 저장 이름 또는 별도 버전 네임스페이스를 사용한다.
- 1.2 저장 가져오기는 자동으로 추론하지 않는다. 가져오기 범위와 손실 규칙이 문서화된 뒤 명시적 사용자 선택으로만 제공한다.
- Phase 0~9의 개발 실패가 사용자의 기존 저장을 변경하지 않아야 한다.

## 11. 그래픽, 오디오와 플랫폼 범위

- 기존 쿼터뷰 던전, 초상, 몬스터, 적, 시설, 효과음과 음악은 보존·재사용한다.
- 경로·배치·Encounter의 재미가 검증되기 전에 신규 최종 그래픽을 제작하지 않는다.
- 임시 표시는 런타임 도형 또는 기존 저장소 자산으로 만든다.
- 새 그래픽이 실제로 필요해지면 GPT 내부 이미지 생성 모델만 사용하고 `assets/source/imagegen/<asset>/SOURCE.md` 출처 계약을 지킨다.
- DAY 1~5 재구축은 PC를 먼저 완료한다.
- 모바일 UI 전면 개편은 PC 버티컬 슬라이스와 판매 가능성 게이트 뒤로 연기한다.
- Web 데모 산출물은 `test/web-*` 정책을 따르며 소스 브랜치에 커밋하지 않는다.

## 13. Phase 공통 PR 계약

각 PR 본문과 핸드오프에는 아래를 적는다.

1. 목표 버전, 기준 SHA, 대상 Phase와 제품 결정
2. 변경 전 실패 또는 부재를 증명하는 관련 테스트
3. 포함 파일과 의도적으로 제외한 파일
4. 사용자에게 달라지는 선택과 화면
5. 데이터·저장 호환성
6. 실행한 관련 테스트와 UI 확인
7. 전체 회귀·전체 플레이·검수 에이전트 요청 여부
8. 미해결 위험과 다음 Phase 진입 조건
9. `Review task ID`, `Reviewed SHA`, `Review range`, `Remaining P1/P2`, `Final review result`

전체 검수가 요청되지 않은 Phase는 정책 필드를 `NOT_REQUESTED / N/A / TARGETED_PASS`로 기록한다. 자동 테스트 결과를 재미 PASS로 표현하지 않는다.

## 14. 금지 범위

- 기존 화면의 색과 크기만 바꿔 UI 전면 개편으로 기록하지 않는다.
- 버튼이 많은 문제를 툴팁 추가로 해결하지 않는다.
- DAY 1~5 게이트 전 새 적, 스토리, 엔딩과 확장 기능을 추가하지 않는다.
- 시설에 공격력·방어력 숫자만 추가하고 전략 시설로 기록하지 않는다.
- HP 배율과 생성 지연만으로 난이도를 조정하지 않는다.
- `GameRoot.gd`에 새 v2 상태와 조건을 계속 누적하지 않는다.
- PC와 모바일 전면 개편을 동시에 진행하지 않는다.
- 자동 회귀, 자동 대리, 다중 seed를 사람 재미 검증으로 기록하지 않는다.
- 경로와 시설 재미가 검증되기 전 신규 최종 그래픽을 만들지 않는다.
- v1.2 공개 태그, Release, 저장 파일과 원격 과거 이력을 수정하지 않는다.
