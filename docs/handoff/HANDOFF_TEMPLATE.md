## 6. 테스트 및 검수

| 순서 | 검수 명령 또는 방법 | 결과 | 근거 경로 |
|---:|---|---|---|
| 1 | 관련 테스트 | PASS/FAIL | 로그 또는 캡처 |
| 2 | 전체 회귀 테스트 | REQUESTED/PASS/FAIL/NOT_REQUESTED | 로그 또는 보고서 |
| 3 | 시각/실플레이 검수 | REQUESTED/PASS/FAIL/NOT_REQUESTED | 캡처 또는 기록 |

전체 회귀, 전체 플레이와 검수 에이전트는 사용자가 현재 작업에서 명시적으로 요청했을 때만 실행한다.

### 검수 에이전트 반복 기록

| 회차 | 검수 작업 ID | 검수 범위 (`base..head`) | 대상 최종 SHA | 주요 지적 | 수정 내용 | 근거 경로 | 재검수 결과 |
|---:|---|---|---|---|---|---|---|
| 1 | 작업/에이전트 ID | `base..head` | 40자리 SHA | 내용 | 내용 | 보고서/로그 | PASS/FAIL |

- 남은 P1/P2 지적:
- 실행하지 못한 필수 검수와 이유:
- PASS 이후 기능·데이터·자산 변경 여부:

검수 PASS는 표에 적은 대상 최종 SHA에만 유효하다. 그 이후 기능, 데이터 또는 자산이 바뀌면 회차를 추가하고 재검수한다.

### 정책 CI용 최종 승인 필드

아래 영문 키는 자동 검사가 읽으므로 번역하거나 이름을 바꾸지 않는다. 전체 검수를 요청받지 않은 기본 작업은 `NOT_REQUESTED / N/A / TARGETED_PASS` 조합을 사용한다. 사용자가 요청한 전체 검수를 완료한 경우에만 실제 작업 ID / 0 / PASS 조합을 사용한다.

- Review task ID:
- Reviewed SHA:
- Review range:
- Remaining P1/P2:
- Final review result:

코드, 데이터, 자산, 씬, 도구 또는 워크플로를 변경한 PR은 이 다섯 필드를 모두 채운 날짜별 핸드오프를 추가한다. 검수 기준 SHA 이후에는 `docs/handoff/` 파일만 변경할 수 있다. `Review range`는 축약명 대신 `40자리 기준 SHA..40자리 Reviewed SHA` 형식으로 기록하며 기준 SHA는 정책 CI가 계산한 merge base와 일치해야 한다.
