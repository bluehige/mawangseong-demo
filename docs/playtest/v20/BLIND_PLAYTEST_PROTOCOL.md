# 제품 2.0 DAY 1~5 블라인드 플레이테스트 프로토콜

## 1. 목적과 표본

- 대상: 이 게임의 2.0 DAY 1~5를 처음 보는 실제 사람 6~10명
- 제외: 개발 참여자, 플레이 방법을 미리 들은 사람, 기존 2.0 빌드를 플레이한 사람
- 기본 환경: PC, 키보드·마우스, 1280×720 이상, 동일한 build SHA
- 관찰 종료: DAY 3 도달과 첫 패배 후 재도전까지. 먼저 일어나는 것만 보고 끝내지 않으며 둘 다 관찰한다. DAY 3까지 패배가 없으면 설명 없이 DAY 5까지 계속한다.

## 2. 시작 전 준비

1. `git rev-parse HEAD` 또는 배포 artifact의 source SHA가 `RESULTS.json`의 `build_sha`와 같은지 확인한다.
2. 이전 참가자의 `user://v20/campaign_v20.json`을 보관할 필요가 없으면 게임 UI의 새 시작으로 덮어쓴다. 1.2 저장은 삭제하지 않는다.
3. 참가자에게 익명 ID를 배정하고 동의·첫 플레이·사전 설명 없음 세 항목을 확인한다.
4. 진행자는 별도 타이머와 관찰지를 준비한다. 게임 화면의 2.0 진입 버튼을 대신 눌러주지 않는다.

## 3. 진행자가 읽을 문장

다음 문장만 그대로 읽는다.

> 지금부터 이 게임을 처음 실행한다고 생각하고 DAY 1부터 플레이해 주세요. 화면에서 알 수 있는 정보만 사용해 자유롭게 진행해 주세요. 무엇을 보고 어떤 결정을 하는지는 소리 내어 말해도 좋습니다. 조작이 완전히 불가능한 기술 문제만 알려 주세요.

게임 규칙, 추천 난이도, 좋은 시설, 몬스터 역할, 경로, 명령, 승리법은 설명하지 않는다.

## 4. 실시간 관찰

진행자는 아래 사건의 시각과 사실만 적는다.

- 2.0 화면이 보인 시점부터 첫 의미 있는 선택까지의 초
- 첫 의미 있는 선택 종류: 시설 설치·교체, 몬스터 배치, 교리 선택
- 참가자가 도움을 요청한 횟수와 진행자가 실제 게임 설명을 한 횟수
- DAY 1 완료 여부
- DAY 3 도달 여부
- 참가자가 시설·배치가 결과를 바꿨다고 스스로 말한 최초 시점과 원문
- 첫 패배 여부, 패배 뒤 바꾸겠다고 말한 내용, 실제 재도전 여부

기술 문제로 진행자가 개입하면 `external_help_count`에 포함하지 않되 `technical_incident_notes`에 적는다. 게임 규칙이나 다음 행동을 알려주면 반드시 `external_help_count`를 올린다.

## 5. 중립 질문

### DAY 1 직후

- “방금 무엇을 지키려고 했나요?”
- “어떤 선택이 결과에 영향을 줬다고 생각하나요?”

정답을 유도하거나 선택지를 제시하지 않는다. DAY 1 완료는 게임 상태로만 판정한다.

### DAY 3 도달 직후

- “시설이나 몬스터 배치가 결과를 어떻게 바꿨나요? 본인 말로 설명해 주세요.”

시설·배치와 결과 사이의 인과를 참가자가 스스로 말해야 `explained_effect_by_day_three=true`다. “그냥 세졌다”, 진행자의 문장을 반복한 답, 빈 답은 false다.

### 첫 패배 직후

- “다음 시도에서 무엇을 한 가지 바꾸고 싶나요?”
- 답을 들은 뒤 “다시 시도하시겠어요?”

한 가지 이상의 구체적 변경을 자기 말로 말하고 실제 재도전 버튼을 눌러야 `failure_change_and_retry=true`다. 패배가 없으면 이 기준은 보수적으로 미충족으로 기록한다.

## 6. 사후 질문

`PARTICIPANT_FORM.md`의 재미·이해도·피로도 1~5점과 자유 응답을 받는다. 이 점수는 정성 판단 자료이며 자동 `GO` 판정에는 사용하지 않는다.

## 7. 세 가지 판매 가능성 기준

유효한 참가자 6~10명에서 아래를 모두 만족해야 `GO`다.

1. 80% 이상이 외부 게임 설명 없이 DAY 1 완료
2. 70% 이상이 DAY 3까지 시설 또는 배치가 결과를 바꿨다고 자기 말로 설명
3. 70% 이상이 실제 패배 뒤 바꿀 점을 하나 이상 말하고 재도전

첫 의미 있는 선택 90초 이내 비율은 별도 핵심 진단으로 함께 보고하지만, 위 세 가지를 임의로 대체하지 않는다. 표본 부족은 `PENDING`, 기록 형식·동의·첫 플레이 조건 위반은 `INVALID`, 기준 미달은 `NO_GO`다.

## 8. 중단과 예외

- 참가자는 언제든 이유 없이 중단할 수 있다. 중단 기록은 원본에 남기되 강요하지 않는다.
- 멀미, 불쾌감, 접근성 문제는 즉시 세션을 멈추고 익명 메모만 남긴다.
- 크래시·입력 불능처럼 게임을 평가할 수 없는 기술 사고는 해당 세션을 유효 표본으로 세지 않고 동일 원인을 수정한 새 build SHA로 cohort를 다시 시작한다.
- 서로 다른 build SHA의 참가자를 한 cohort에 섞지 않는다.
