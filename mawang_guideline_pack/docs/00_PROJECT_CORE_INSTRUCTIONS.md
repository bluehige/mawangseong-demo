# 00. 프로젝트 핵심 지침

프로젝트명: **마왕님, 마왕성은 누가 지켜요?**
문서 목적: 새 기획 세션, Codex 구현 세션, 이미지 리소스 제작 세션, QA 세션에 공통으로 적용할 기준 문서.

이 문서는 프로젝트의 “헌법”이다. 다른 세션에서 세부 구현이 바뀌더라도 아래의 핵심 방향은 유지한다. 변경이 필요하면 별도 변경 요청으로 기록한다.

---

## 1. 확정된 핵심 방향

| 항목 | 확정 내용 |
|---|---|
| 장르 | 마왕성 건설/운영 + 몬스터 육성 + 실시간 전투 디펜스 |
| 주인공 | F급 신입 마왕 |
| 배경 | 초라한 동굴을 진짜 마왕성으로 성장시키는 큐트 호러 판타지 |
| 데모 목적 | 기능 구현 검증용 초소형 플레이어블 데모 |
| 엔진 | Godot 4.5 |
| 언어 | GDScript |
| 화면 기준 | PC 16:9, 1920×1080 |
| 시점 | 2D 쿼터뷰 |
| 그래픽 톤 | 큐트 호러, 어둡지만 귀여움, 보라색 마력 조명, 동굴 마왕성 분위기 |
| 입력 | 마우스 중심. 클릭 선택, 우클릭 이동, UI 버튼 지휘 |
| 마왕성 구조 | 데모는 슬롯형 방 구조. 추후 자유건설 확장 가능하게 데이터 구조 설계 |
| 데모 길이 | 3일, 10~15분 |
| 초기 몬스터 | 슬라임, 고블린, 임프 |
| 초기 적 | 탐험가, 도적, 견습 용사 |
| 행동지침 | 전체 지침 + 방 단위 지침 |
| 직접 조종 | 선택한 몬스터 1마리 직접 조종 가능 |
| 성인 요소 | 없음 |
| 잔혹 표현 | 직접적 고어 금지. 전투불능/퇴각 중심 |
| 원작 참고 방식 | 구조적 레퍼런스만 참고. 캐릭터, UI, 명칭, 수치, 이벤트는 오리지널 |

## 2. 게임의 핵심 판타지

플레이어는 막 임명된 신입 마왕이다. 하지만 받은 것은 웅장한 성이 아니라 낡은 동굴이다. 인간 왕국은 이곳을 만만하게 보고 탐험가, 도적, 견습 용사를 보낸다. 플레이어는 슬라임, 고블린, 임프 같은 약한 부하들을 배치하고 지휘해 침입자를 막으며, 동굴을 점차 진짜 마왕성으로 키운다.

핵심 감정은 다음이다.

1. **허술한 시작**: “이게 마왕성이라고?”라는 코믹한 출발.
2. **부하 애착**: 약한 몬스터들이 명령에 따라 싸우고 성장한다.
3. **전술 개입**: 자동 전투를 보다가 중요한 순간 직접 지휘한다.
4. **성장 체감**: 동굴이 점점 마왕성다워진다.

## 3. 데모의 핵심 검증 질문

데모에서 검증할 것은 콘텐츠 양이 아니라 재미의 뼈대다.

| 검증 질문 | 통과 기준 |
|---|---|
| 전투가 읽히는가? | 몬스터, 적, 방, 경로, 함정이 즉시 구분된다 |
| 행동지침이 의미 있는가? | 사수/총공격/생존 우선/함정 유도에 따라 결과가 달라진다 |
| 직접 조종이 재미있는가? | 한 몬스터 조작으로 전황을 바꿀 수 있다 |
| 운영 루프가 돌아가는가? | 관리 → 전투 → 결산 → 다음 날 흐름이 끊기지 않는다 |
| 마왕성 방어 판타지가 살아있는가? | 왕좌의 방, 보물고, 가시 복도 방어가 목적처럼 느껴진다 |

## 4. 반드시 유지할 디자인 기둥

### 4-1. 마왕성은 단순 맵이 아니라 “방어해야 할 집”이다

방은 단순 배경이 아니다. 각 방은 기능과 위험을 가진다.

- 입구: 침입자 등장 지점.
- 가시 복도: 함정/지연 구역.
- 병영: 부하 몬스터 배치/대기.
- 보물 보관실: 도적이 노리는 목표.
- 회복 둥지: 부상 몬스터 후퇴 지점.
- 왕좌의 방: 최종 방어 목표.

### 4-2. 몬스터는 소모품이 아니라 부하다

몬스터는 이름, 역할, HP, 공격력, 방어력, 이동속도, 지능, 충성도, 스킬 슬롯을 가진다. 데모에서는 성장 시스템을 최소화해도, 구조는 성장 확장을 전제로 둔다.

### 4-3. 전투는 자동이지만 방치가 아니다

몬스터는 AI로 움직인다. 플레이어는 전투 중 다음 방식으로 개입한다.

- 전체 지침 변경.
- 방 단위 지침 변경.
- 선택 몬스터 직접 조종.
- 스킬 사용.

### 4-4. 악명은 마왕 게임의 핵심 자원이다

데모에서는 악명 기능을 단순 표시/결산 수준으로 구현해도 된다. 정식 확장에서는 악명이 인간 침공 강도와 보상 규모에 영향을 준다.

## 5. 데모에 넣을 것과 넣지 않을 것

### 데모에 반드시 넣을 것

1. 3일 진행.
2. 쿼터뷰 마왕성 맵.
3. 몬스터 3종: 슬라임, 고블린, 임프.
4. 적 3종: 탐험가, 도적, 견습 용사.
5. 자동 전투.
6. 전체 지침: 사수, 총공격, 생존 우선.
7. 방 지침: 입구 봉쇄, 함정 유도, 후퇴선 유지.
8. 직접 조종: 선택 몬스터 1마리.
9. 임프 화염구 투사체.
10. 가시 복도 함정.
11. 왕좌의 방 HP.
12. 전투 로그.
13. 결산 화면.

### 데모에서 제외할 것

1. 자유건설 전체 시스템.
2. 대규모 스토리.
3. 장비 시스템.
4. 진화 시스템.
5. 특수 몬스터.
6. 세력별 침공.
7. 영구 사망.
8. 복잡한 속성 상성.
9. 확률형 뽑기.
10. 성인 요소.

제외 요소는 “삭제”가 아니라 “후순위”다. 데이터 구조는 확장 가능하게 만든다.

## 6. 기술 지침

### 6-1. 기본 구조

- Godot 4.5.
- GDScript.
- 2D 쿼터뷰.
- 쿼터뷰는 2D 기반으로 구현한다. 3D 씬 전환이 아니라 `Node2D`, `CharacterBody2D`, Y-sort, 발밑 피벗, 깊이감 있는 배경/오브젝트 배치로 연출한다.
- `CharacterBody2D` 기반 유닛.
- `TileMapLayer` 기반 맵 레이어.
- `CanvasLayer` 기반 HUD.
- JSON 기반 데이터 로딩.
- Autoload: `GameState`, `SignalBus`, 필요 시 `DataRegistry`.

### 6-1-A. 쿼터뷰 구현 기준

- 이동 좌표와 방 경로는 기존 2D 좌표계를 유지한다.
- 유닛의 기준점은 발밑 중앙이다. 충돌체와 선택 판정도 발밑 주변에 작게 둔다.
- `z_index` 또는 Y-sort 기준은 화면 y값을 우선 사용해 앞/뒤 겹침이 자연스럽게 보이게 한다.
- 벽, 기둥, 문틀, 방 장식은 유닛을 가릴 수 있으므로 전투 가독성을 해치지 않는 높이와 투명도를 유지한다.
- 방 배경은 완전 평면 바닥이 아니라 뒤쪽 벽/문/소품이 보이는 3/4 하향 시점으로 제작한다.
- UI는 쿼터뷰 맵 위를 과하게 덮지 않는다. 전투 중 유닛, 경로, 함정, 왕좌, 보물방이 항상 읽혀야 한다.

### 6-2. 기본 씬 구조

```text
GameRoot(Node)
  ManagementScene(Node2D)
  CombatScene(Node2D)
  HUD(CanvasLayer)
  ResultPanel(CanvasLayer)
```

```text
CombatScene(Node2D)
  DungeonMap(Node2D)
    GroundLayer(TileMapLayer)
    WallLayer(TileMapLayer)
    DecorLayer(TileMapLayer)
    RoomNodes(Node2D)
    TrapNodes(Node2D)
  Units(Node2D)
  Projectiles(Node2D)
  Effects(Node2D)
  Camera2D
  CombatManager(Node)
  WaveManager(Node)
```

### 6-3. 데이터 주도 설계

몬스터, 적, 스킬, 방, 웨이브는 하드코딩하지 않는다. 초기에는 GDScript 상수로 시작해도 되지만, 최종 구조는 `data/*.json` 로드 방식으로 전환 가능해야 한다.

```text
data/
  monsters.json
  enemies.json
  skills.json
  rooms.json
  waves.json
```

### 6-4. 명명 규칙

- 파일명: snake_case.
- 씬명: PascalCase.
- 클래스명: PascalCase.
- 변수/함수: snake_case.
- 신호명: snake_case.
- 데이터 ID: 영문 snake_case.

예시:

```text
monster_slime
monster_goblin
monster_imp
enemy_explorer
enemy_thief
enemy_trainee_hero
room_spike_corridor
skill_fireball
```

## 7. 전투 기준

### 7-1. 기본 전투 공식

```text
최종 피해 = max(1, 공격력 - 방어력 × 0.5)
```

데모에서는 치명타, 명중률, 속성 상성은 제외한다.

### 7-2. 유닛 상태

```text
IDLE
MOVE_TO_ROOM
SEEK_TARGET
MOVE_TO_TARGET
ATTACK
CAST_SKILL
RETREAT
DIRECT_CONTROL
STUNNED
DOWN
```

### 7-3. 타깃 우선순위

1. 현재 방 안의 적.
2. 왕좌의 방에 가까운 적.
3. 체력이 낮은 적.
4. 가장 가까운 적.

도적은 예외적으로 보물 보관실을 우선 목표로 한다. 견습 용사는 왕좌의 방을 우선 목표로 한다.

## 8. UI 기준

### 8-1. 전투 화면

| 위치 | 구성 |
|---|---|
| 상단 | 금화, 마력, 식량, 악명, 날짜, 마왕성 체력 |
| 좌측 | 미니맵, 방 목록, 전투 로그 |
| 중앙 | 쿼터뷰 마왕성 전투 맵 |
| 우측 | 선택 유닛 정보 |
| 하단 | 전체 지침/방 지침/직접 조종 버튼 |

### 8-2. 관리 화면

필수 버튼:

- 건설.
- 몬스터.
- 침공 작전. 데모에서는 비활성 가능.
- 방어 준비.
- 다음 날.

### 8-3. 몬스터 관리 화면

필수 표시:

- 몬스터 카드 목록.
- 선택 몬스터 상세 능력치.
- 스킬 슬롯 3칸.
- 충성도.
- 훈련/배치/스킬 교체/돌아가기 버튼.

## 9. 그래픽 리소스 기준

### 9-1. 캐릭터

| 항목 | 기준 |
|---|---|
| 스타일 | 큐트 호러 SD 쿼터뷰 |
| 파일 | PNG 투명 배경 |
| 기준 캔버스 | 128×128px per frame |
| 피벗 | 발밑 중앙 |
| 방향 | front/down, back/up, side, side_flip |
| 애니메이션 | idle, walk, attack, cast, hit, down |

### 9-2. 타일

| 항목 | 기준 |
|---|---|
| 타일 크기 | 64×64px |
| 바닥 | 동굴 바닥 |
| 벽 | 동굴/돌벽 |
| 장식 | 보라 횃불, 해골, 작은 깃발, 마력 수정 |
| 오브젝트 | 왕좌, 보물더미, 병영 무기대, 회복 둥지, 가시 함정 |

### 9-3. 중요한 원칙

UI 텍스트는 이미지에 굽지 않는다. 모든 텍스트는 Godot Label로 표시한다. 이미지 리소스에는 텍스트 없는 아이콘과 오브젝트만 만든다.

쿼터뷰 리소스는 바닥만 내려다보는 평면 구도가 아니라 앞면/측면/상단이 조금씩 보이는 각도를 기본으로 한다. 다만 충돌, 선택, 이동 목적지는 항상 발밑 기준으로 맞춘다.

## 10. 변경 관리 원칙

새 아이디어가 나와도 즉시 구현하지 않는다. 다음 기준으로 분류한다.

| 분류 | 처리 |
|---|---|
| 데모 핵심 | 즉시 구현 후보 |
| 데모 보조 | 일정 여유가 있을 때 구현 |
| 확장판 기능 | 문서화만 하고 구현 보류 |
| 방향 충돌 | 결정 로그에 기록 후 별도 판단 |

특히 아래 요소는 데모 이후로 미룬다.

- 자유건설.
- 진화.
- 특수 몬스터 5스킬 슬롯.
- 세력별 악명.
- 다층 마왕성.
- 대규모 침공.

---

## 11. 세션 작업 원칙

1. 한 세션에는 하나의 기능 묶음만 다룬다.
2. 산출물은 항상 “구현 가능”해야 한다.
3. 기획만 늘리지 말고 완료 조건을 적는다.
4. Placeholder 그래픽으로 먼저 구현한다.
5. 코드가 돌아가면 그다음 리소스를 교체한다.
6. 모든 변경은 결정 로그에 남긴다.
7. 새 기능은 기존 데모 루프를 깨지 않아야 한다.

---

## 12. 최종 데모 완료 조건

- 관리 화면에서 시작한다.
- 방어 준비 후 전투 화면으로 전환된다.
- 적이 입구에서 등장해 목표 방으로 이동한다.
- 몬스터가 자동으로 적을 탐색하고 공격한다.
- 전체 지침이 전투 행동에 영향을 준다.
- 방 지침이 특정 방 행동에 영향을 준다.
- 선택 몬스터 직접 조종이 가능하다.
- 임프 화염구가 투사체로 발사된다.
- 가시 복도 함정이 피해/둔화를 준다.
- 왕좌의 방 HP가 0이 되면 패배한다.
- 3일차 견습 용사를 격퇴하면 클리어된다.
- 결산 화면이 표시된다.
