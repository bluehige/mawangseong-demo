## 1. 현재 고정 결정

- 프로젝트명은 「마왕님, 마왕성은 누가 지켜요?」다.
- 엔진은 Godot 4.5, 언어는 GDScript다.
- 화면 기준은 PC 16:9, 1920x1080이다.
- 전투/관리 맵 목표 시점은 2D 쿼터뷰다.
- 쿼터뷰는 완전 3D가 아니라 2D 좌표계, 발밑 피벗, Y-sort, 쿼터뷰 스프라이트/배경으로 구현한다.
- 데모는 3일짜리 기능 테스트다.
- 몬스터는 슬라임, 고블린, 임프 3종을 기준으로 한다.
- 적은 탐험가, 도적, 견습 용사 3종을 기준으로 한다.
- 전투는 자동 전투를 기본으로 하고, 전체 지침, 방 지침, 선택 몬스터 직접 조종으로 개입한다.

## 2. 쿼터뷰 구현 기준

- 이동 경로, 방 연결, 충돌, 클릭 선택은 기존 2D 좌표계를 사용한다.
- 유닛의 기준점은 발밑 중앙이다.
- 충돌체는 몸 전체가 아니라 발밑 주변에 작게 둔다.
- `z_index` 또는 Y-sort는 화면 y값을 우선 사용한다.
- 방 배경은 바닥, 뒤쪽 벽, 출입구, 주요 소품이 보이는 3/4 하향 구도를 사용한다.
- 벽과 소품은 분위기를 만들되 유닛, 경로, 함정, 왕좌, 보물방을 가리면 안 된다.
- 앞쪽 오브젝트가 유닛을 가릴 경우 투명도, 높이, z_index를 조정한다.
- UI는 맵의 깊이감을 해치지 않되 전투 판독을 우선한다.

## 3. 이미지 생성 기준

- 캐릭터는 SD/chibi 비율, 투명 배경, 128x128 프레임 기준이다.
- 프롬프트에는 `3/4 overhead quarter-view game sprite`를 명시한다.
- 얼굴, 몸통, 발밑 기준점이 모두 읽혀야 한다.
- 방향은 `front/down`, `back/up`, `side`, `side_flip` 기준으로 잡는다.
- 방/오브젝트는 앞면과 측면이 조금 보이는 쿼터뷰 구도를 사용한다.
- 이미지 안에는 UI 텍스트를 넣지 않는다. 텍스트는 Godot Label로 처리한다.

## 4. QA 기준

- 방의 바닥/벽/출입구/소품이 쿼터뷰로 읽히는지 확인한다.
- 선택 링, HP 바, 충돌체, 이동 목표가 발밑 기준으로 맞는지 확인한다.
- 앞뒤 유닛 정렬이 화면 y값 기준으로 자연스러운지 확인한다.
- 장식이 유닛과 핵심 UI를 가리지 않는지 확인한다.
- 1일차는 쉽게 승리, 2일차는 방 지침이 유리, 3일차는 직접 조종/임프 스킬이 안정 승리에 필요해야 한다.

## 5. 세션 운영 기준

- 한 세션에는 하나의 기능 묶음만 다룬다.
- 먼저 실행 가능한 최소 구현을 만든다.
- 새 아이디어는 데모 핵심, 데모 보조, 확장판 후보, 방향 충돌로 분류한다.
- 방향 충돌은 바로 구현하지 말고 결정 로그에 남긴다.
- 모든 구현 세션은 실행/테스트 명령과 결과를 남긴다.

## 6. 핸드오프 작성 기준

세션을 넘길 때는 핸드오프 문서와 작업 로그를 분리한다.

핸드오프 문서에는 다음을 반드시 쓴다.

1. 현재 목표와 최신 결정
2. 이번 세션에서 변경한 파일
3. 구현 완료 기능
4. 검증한 명령과 결과
5. 미완성/보류 항목
6. 다음 세션 첫 작업
7. 주의할 리스크와 건드리면 안 되는 파일
8. 새 세션에 그대로 붙여 넣을 시작 문장

작업 로그에는 다음을 쓴다.

1. 사용자 요청
2. 작업 전 판단
3. 실제 작업 내용
4. 검증 결과
5. 남은 판단 사항
